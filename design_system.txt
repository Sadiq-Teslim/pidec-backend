1.   About This Design System
This is the interim design system for the PIDEC 1.0 platform. It is grounded in the official colours of the University of Lagos Faculty of Engineering and designed to be consistent, accessible, and production-ready. It will serve as the authoritative reference for all frontend development until the official design system from the design lead is delivered.
Everything in this document — tokens, typography, components, spacing — is a binding standard. No component library other than Mantine is to be used. No font outside of Plus Jakarta Sans and DM Sans is to be used. Deviations require explicit approval from the Technical Lead.
2.   Brand Identity
PIDEC 1.0 is the first edition of a faculty-wide engineering competition at the University of Lagos. The visual identity is rooted in the Faculty of Engineering’s official colours — navy blue and white — extended with gold as a prestige accent. The tone is professional, credible, and modern. It should feel like a serious competition platform, not a student club.

3.   Colour System
The PIDEC colour system is built on three primary roles: Navy (authority, structure), White (space, clarity), and Gold (prestige, action). Supporting shades are derived from these primaries by adjusting lightness only — the hue never changes.

4.   Typography
PIDEC uses two fonts exclusively. Plus Jakarta Sans for all display, heading, and hero text. DM Sans for all body copy, UI labels, inputs, captions, and data. Both are Google Fonts and loaded via next/font for zero layout shift.

5.   Spacing System
PIDEC uses an 8px base grid. All spacing values — margins, paddings, gaps — must be multiples of 8. Mantine’s spacing scale maps directly to this. Never use arbitrary pixel values for spacing.
6.   Layout & Grid
The PIDEC platform uses a 12-column grid with fluid gutters. Mantine’s Container and Grid components enforce this automatically.
7.   Component System (Mantine)
All UI components come from Mantine. No other component library is to be imported. Custom components must be built on top of Mantine primitives — never from scratch using raw HTML/CSS where a Mantine component exists. All Mantine components are themed globally via the MantineProvider theme object.
7.1   Global Mantine Theme Config
The theme is configured once in src/app/providers.tsx and applies globally. Key overrides:
primaryColor: 'navy'
primaryShade: { light: 8, dark: 7 }
fontFamily: 'DM Sans, sans-serif'
headings: { fontFamily: 'Plus Jakarta Sans, sans-serif' }
defaultRadius: 'md'  // 8px
colors: { navy: [...10 shades], gold: [...10 shades] }
spacing: { xs:4, sm:8, md:16, lg:24, xl:32 }  // maps to --space tokens
breakpoints: { xs:'36em', sm:'48em', md:'62em', lg:'75em', xl:'88em' }

8.   Iconography
All icons come from the Tabler Icons library (@tabler/icons-react). This is the default icon set that ships with Mantine. No other icon set is to be used. Icons are always rendered at a consistent size per context.
9.   Accessibility
PIDEC targets WCAG 2.1 Level AA compliance. This is not optional. The platform will be used by 3,000+ students across a range of devices and conditions. Every component must be accessible by default.
10.   Frontend File Structure (FSD)
The frontend follows Feature-Sliced Design (FSD). Every piece of code lives in a defined layer. Nothing imports from a higher layer. Shared utilities never import from features. This keeps the codebase navigable and scalable.

Button Rules
1. Never use more than one Primary button per view section.
2. Always pair a primary button with an outline or subtle secondary.
3. All buttons must have a loading state using Mantine's loading prop.
4. Danger buttons require a confirmation modal before action executes.
5. Disabled state must always visually communicate non-interactability — use opacity 0.5.

--TABLE 17--
Form Rules
1. All forms use Mantine's useForm hook with Zod schema validation.
2. Error messages appear inline below each field — never as a toast.
3. All forms have a loading/disabled state on submit while awaiting response.
4. Required fields are marked with asterisk in label via Mantine's withAsterisk.
5. File inputs validate type and size client-side before triggering any upload.
6. Word count helper is shown live for all Textarea fields with word limits.

Icon Rules
1. Never use emoji as icons in UI components.
2. Icon + text: icon always on the left, 8px gap.
3. Decorative icons have aria-hidden='true'.
4. Interactive icon-only buttons have aria-label and a Tooltip.

--TABLE 22--
Requirement | Standard
Colour contrast (text) | 4.5:1 minimum for normal text. 3:1 for large text (18px+ bold or 24px+).
Colour contrast (UI) | 3:1 for interactive UI components (buttons, inputs, focus rings).
Focus rings | All interactive elements must have a visible focus ring. Use Mantine's default focus ring — do not suppress it.

Use DM Sans for all body and UI text. | Use font weights outside 400, 500, 600 (DM) and 700, 800 (PJS).

Validate all forms with Zod schemas. | Let invalid data reach the API layer.
Import only named Tabler icons. | Import entire icon libraries or use emoji as icons.
Show a Skeleton while data loads. | Show an empty screen or a spinner that replaces content.

Add aria-label to all icon-only buttons. | Leave interactive elements without accessible labels.
Use CSS variables for all colour values. | Hardcode hex values in component styles.
Test at 360px mobile width first. | Design desktop-first and bolt on mobile later.
Confirm before all destructive actions. | Execute deletes or disqualifications on a single click.
